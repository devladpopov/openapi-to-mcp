## КРИТИЧЕСКОЕ: поведение после compaction / при "Continue"

Claude Code умеет автоматически сжимать историю диалога, когда
контекстное окно переполняется, и вставляет инструкцию
"Continue from where you left off". Это НЕ команда повторить
уже выполненную работу — это просто шаблонная подсказка,
оставшаяся после сжатия.

### Порядок действий при compaction / получении "Continue":

1. **Прочитай `CHECKPOINT.md`** в корне проекта (если есть).
   - Если `STATUS: COMPLETED` — НЕ продолжай работу.
     Ответь коротко: "по <задаче> всё готово, жду новых
     инструкций" и остановись.
   - Если `STATUS: IN_PROGRESS` — продолжай с поля `NEXT:`,
     но СНАЧАЛА верифицируй, что пункты из `DO_NOT_REDO:`
     уже сделаны (через `git log`, `curl /health`, список
     миграций и т.п.).

2. **Проверь TodoList.** Если все items completed и нового
   запроса от пользователя нет — ничего не делай, просто
   подтверди завершение.

3. **Верифицируй перед любым деплоем / миграцией.** Перед
   destructive-действием обязательно проверь текущее
   состояние: release version, последний коммит, статус
   миграций. Если реальность уже соответствует цели —
   пропусти шаг и залогируй "already done".

4. **Не дублируй tool calls.** Если в истории был идентичный
   успешный tool call — не повторяй без явной новой причины.

### Как писать CHECKPOINT.md

По завершении любой крупной задачи (деплой, миграция,
фича-ветка, настройка сервиса) — перезапиши `CHECKPOINT.md`
в корне проекта:

```
STATUS: COMPLETED
TASK: краткое описание того, что было сделано
DATE: 2026-04-23
LAST_ACTION: что было сделано самым последним
NEXT: await user input
DO_NOT_REDO:
  - пункт 1 (что уже идемпотентно сделано)
  - пункт 2
VERIFY_BEFORE_ACT:
  - команды для проверки (git log -1, curl /version, etc.)
```

При старте новой задачи — поставь `STATUS: IN_PROGRESS` и
обновляй `NEXT:` по ходу работы.

## Молчание между tool calls

Роутер шлёт в Telegram КАЖДЫЙ текстовый блок между tool calls
как отдельное сообщение. Не комментируй каждый шаг —
пользователь получит простыню "Сейчас проверю / Деплою /
Готово, проверю статус" отдельными сообщениями.

- Делай tool calls подряд, без промежуточных комментариев.
- Пиши один финальный ответ по результату.
- Исключение: задача длинная (>30 сек или >5 tool calls) —
  можно один промежуточный апдейт "работаю, ещё ~N шагов".

## КРИТИЧЕСКОЕ ОГРАНИЧЕНИЕ: запрет на самоизменение роутера

**Причина:** бот работает внутри bun-процесса. Если редактировать
код роутера и перезапускать bun — claude.exe (наш процесс) тоже
умрёт. Контекст беседы потеряется, файл может записаться не до конца,
роутер поднимется с битым кодом.

### Что НЕЛЬЗЯ делать:
- Редактировать файлы в `claude-topic-router/src/` или `claude-topic-router/runner/`
- Убивать процессы bun, node, claude
- Перезапускать роутер или runner
- Менять конфигурацию `config/settings.json`

### Что НУЖНО делать вместо этого:
Если пользователь просит изменить поведение бота, форматирование,
роутер или runner — **предложить решение и передать в Cowork:**

1. **Проанализировать проблему** — прочитать нужные файлы, найти причину
2. **Описать решение** — какие файлы менять, что именно исправить
3. **Написать готовый контекст** для Cowork-сессии:

```
Влад, это изменение нужно применить через Cowork.
Скопируйте в Cowork следующее:

---
Задача: [краткое описание]
Файл: [путь к файлу]
Что изменить: [описание изменения или diff]
После изменения: перезапустить роутер командой [команда]
---
```

4. Можно **читать** файлы (`cat`, `head`, `grep`) для диагностики
5. Можно **редактировать** файлы своего топика (SOUL.md, topic-memory.md и т.д.)
